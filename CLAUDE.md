## Project Overview

万象研究所 (Wanxiang Institute) — a mobile-first knowledge card learning platform covering 21 disciplines (economics, psychology, law, astronomy, etc.). Users explore knowledge via cards, learning paths with skill-tree progression, a knowledge graph, daily quizzes, and thought experiments.

## Commands

```bash
npm run dev        # Start Nuxt dev server
npm run build      # Production build (nuxt build)
npm run generate   # Static site generation (nuxt generate)
npm run preview    # Preview production build
```

No test framework is configured. No linter is configured.

## Tech Stack

- **Nuxt 3** (Vue 3 + auto-imports + file-based routing)
- **Tailwind CSS** via `@nuxtjs/tailwindcss` — custom "macaron" color palette defined in `tailwind.config.ts`
- **Supabase** for database (PostgreSQL + RLS) — client initialized in `utils/supabase.ts`
- **D3.js** (d3-force, d3-selection, d3-zoom) for knowledge graph visualization
- **MiniSearch** for client-side search (dev dependency)

## Architecture

### Dual Data Layer

The app has two data sources that coexist:

1. **Supabase database** — primary source for cards. `utils/cards.ts` fetches via Supabase client. Pages use `useAsyncData()` to call these async functions (e.g., `getAllCards()`, `getCardsByCategory()`).

2. **Local JSON files** in `content/` — used for learning paths (`content/paths/*.json`), quizzes (`content/quizzes/*.json`), thought experiments, and some fallback card data. Loaded via `import.meta.glob()`.

All pages that fetch from Supabase use Nuxt's `useAsyncData()` pattern — utility functions are async and return Promises.

### Database Schema

Defined in `supabase/migrations/001_initial_schema.sql`. Key tables:
- `cards` — knowledge cards with JSONB columns `key_data` and `"references"` (quoted because `references` is a PostgreSQL reserved word)
- `categories` — 23 discipline categories
- `paths` / `path_nodes` / `path_edges` — learning path skill trees
- `quizzes` / `thought_experiments` — daily content
- User tables (`user_progress`, `favorites`, `quiz_results`, `user_achievements`) require auth

Seed data is split across `supabase/seed_batch1.sql` through `seed_batch4.sql` and deployed via `scripts/seed-to-supabase.cjs`.

### Page Structure

| Route | Purpose |
|---|---|
| `/` | Home — hero, today's pick, category grid, path previews |
| `/categories` | All 23 categories with card counts |
| `/categories/[category]` | Cards filtered by category |
| `/cards/[slug]` | Individual card detail view |
| `/paths` | Learning paths listing |
| `/paths/[slug]` | Skill tree visualization with node progression |
| `/graph` | D3.js force-directed knowledge graph |
| `/experiment` | Daily quiz / thought experiment |
| `/profile` | User stats and favorites |

### Layout

`layouts/default.vue` provides `AppHeader` (with search toggle) + `AppBottomNav` (mobile navigation). Search is an overlay (`SearchOverlay` component).

### Client-Only State

Favorites and progress are stored in localStorage (`utils/favorites.ts`, `utils/progress.ts`). These functions guard with `if (import.meta.server) return []` to avoid SSR hydration mismatches.

### Environment Variables

Required in `.env`:
- `SUPABASE_URL` — Supabase project URL
- `SUPABASE_ANON_KEY` — public anon key
- `SUPABASE_SERVICE_KEY` — service role key (for admin operations)

Exposed to frontend via `runtimeConfig.public` in `nuxt.config.ts`.

## Key Conventions

- Component directory structure: `components/{Layout,Card,Search}/*.vue` — Vue auto-imports handle resolution
- Type definitions: `types/index.ts` (cards, categories), `types/paths.ts`, `types/quizzes.ts`
- Category colors and icons are defined both in `types/index.ts` (CATEGORIES array) and `tailwind.config.ts` (`category.*` colors) — keep them in sync
- The `macaron` color palette is the primary design system: `macaron-bg`, `macaron-card`, `macaron-border`, `macaron-text`, `macaron-text-secondary`, `macaron-cta`, `macaron-cta-hover`
- Card component classes (`.card-base`, `.badge-*`) are defined in `assets/css/main.css` via `@layer components`
- **No emoji in code.** Never use emoji characters (📖, 🔬, etc.) anywhere in the codebase — components, templates, data files, types, or seed data. Use professional SVG icon libraries instead (e.g., `@nuxt/icon` with Iconify, Heroicons, Lucide). The project already has `@nuxt/icon` installed; use `<Icon name="..." />` for all icon needs.
- **Screenshot cleanup.** Any screenshot images generated by Claude's screenshot tools must be permanently deleted before ending the session. Do not leave them in the project directory or on local disk. Use `rm -f <path>` to erase each file after use.
